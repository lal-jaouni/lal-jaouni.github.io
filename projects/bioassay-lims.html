Bioassay LIMS

A full-stack laboratory information management system built for bioassay experiments. Manages the complete lifecycle from experimental planning through computational image analysis and quantitative reporting.

Role Sole Developer & Architect
Organization Diagnostics R&D
Scale 60,000+ lines of code
Stack FastAPI, React, TypeScript, Numba
Key Result 20-100x image processing speedup

The Problem

Running plate-based bioassay experiments involves a lot of moving parts: designing 96-well plate layouts, tracking reagents, configuring instruments, processing raw images, and generating quantitative results. Before this system, these steps were handled by a patchwork of spreadsheets, standalone scripts, and manual bookkeeping -- each experiment required hours of data wrangling before anyone could look at results.

The image processing piece was especially painful. Aligning spot grids on raw plate images, correcting for instrument-specific artifacts, and extracting per-spot intensities required computational methods that were too slow for interactive use and too fragile for batch processing.

What I Built

Full-Stack LIMS (60K+ LOC)

A complete experiment management system: 157 REST endpoints across 19 FastAPI routers, backed by 246 Pydantic data models and 19 service-layer classes. The React frontend has 27 pages with a 96-well plate editor supporting drag-and-click layout design, multi-tab reagent configuration, and 5 concurrent run sessions with click-to-exclude well selection.

Data lives in a file-native Parquet + JSON platform with atomic writes -- no database server to manage, fully human-inspectable, and handling 90+ active experiment folders. HTTP Basic Auth with role-based access control (viewer, operator, admin) supports multi-user lab teams.

Computational Image Processing Pipeline

The core technical challenge. Each plate image needs precise grid alignment to extract per-spot signal intensities. I built a two-phase grid search: a coarse pass (~3,700 iterations covering translation and rotation) followed by a fine pass (~7,700 iterations at sub-pixel precision). Together, that's over 11,000 evaluations per image.

To make that practical, I optimized the alignment engine with Numba JIT compilation across 8 hot-path functions, achieving 20-100x speedup over pure Python. The pipeline also handles MAD-based cosmic ray detection (8x MAD threshold), truncated-circle ROI masks for different plate geometries, scale-aware processing across multiple binning levels, and per-spot correction factor calibration.

Multi-Instrument Support

Different plate readers have different sensor geometries and gain characteristics. The system handles multiple instrument types with reader-specific calibration, gain corrections, and spot factor adjustments, producing consistent results regardless of which instrument captured the data.

Hardware Integration

Integrated directly with a laboratory fluidic dispenser via a custom serial driver, with step-by-step workflow execution, concurrent instrument control tabs, and operator tracking. The serial I/O runs on a ThreadPoolExecutor to keep the async FastAPI server responsive.

Batch Processing

Multi-select experiment list with continue-on-error semantics, per-experiment progress tracking, and template auto-matching on instrument and plate configuration. A three-stage data import pipeline filters records against plate layout and run exclusions, reporting transparent record counts at each stage.

Why This Matters

Scientific software lives or dies by how well it models the actual workflow of the people using it. The hardest problems weren't algorithmic -- they were understanding why a bench scientist configures a plate layout the way they do, why certain validation steps can't be skipped, and how to surface results in a way that maps to how they think about their experiments. Building for regulated lab environments requires understanding both the science and the compliance constraints that shape every decision.
