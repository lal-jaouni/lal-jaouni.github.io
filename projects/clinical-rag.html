Clinical RAG Assistant

A production-grade retrieval-augmented generation system for clinical Q&A with 4-layer safety guardrails, achieving 0% hallucination rate on evaluation. Designed for healthcare organizations that need trustworthy AI, not just smart AI.

Role Sole Developer & Architect
Duration 8 weeks
Test Coverage 167 tests
Tech Stack Python, FastAPI, PostgreSQL, pgvector, Ollama, Streamlit, PubMedBERT
Key Result 0% hallucination rate with 4-layer safety pipeline
Clinical inference pipeline

The Problem

Open-source RAG systems work great for customer support chatbots. They are fundamentally unsuitable for clinical settings without significant additional engineering. The standard workflow -- retrieve, generate, return -- provides no mechanism to detect or prevent hallucination, no way to verify source attribution, and no confidence calibration for high-stakes questions.

When an LLM generates a plausible-sounding claim about drug dosing or contraindications that isn't grounded in the retrieved sources, the consequences extend beyond a bad user experience. Healthcare organizations building clinical AI tools need:

  • Hallucination detection: Sentence-level grounding verification against retrieved sources
  • Source attribution: Verifiable citations with PMIDs, publication years, and traceable claims
  • Confidence quantification: Explicit uncertainty signals so clinicians know when the system is guessing
  • Domain boundaries: Detection and refusal of out-of-domain queries

What I Built

4-Layer Safety Architecture

Rather than stacking a single guardrail on top of a base RAG pipeline, I implemented four independent safety layers that each catch different failure modes:

Layer 1: Domain-Specific Retrieval uses PubMedBERT embeddings (trained on biomedical literature) combined with hybrid search -- Reciprocal Rank Fusion over pgvector cosine similarity (weight 0.6) and BM25 keyword matching (weight 0.4). This matters because generic embeddings fail on clinical abbreviations and domain-specific terminology. Automatic query expansion via a curated synonym map ensures "MTP" retrieves "massive transfusion protocol."

Layer 2: Source Grounding Verification checks every sentence in the LLM's answer against the retrieved source chunks using embedding cosine similarity. If maximum similarity falls below 0.50, the sentence is flagged as potentially ungrounded. A secondary n-gram overlap check catches semantic similarity mismatches. This prevents the most dangerous hallucination pattern: synthesized information that sounds authoritative.

Layer 3: Confidence Scoring is a composite metric combining citation density (how many claims are backed by sources), grounding score (average embedding similarity between answer and sources), and uncertainty language detection (hedging phrases like "may," "possibly," "unclear"). Answers scoring below 0.70 confidence trigger a hard override: the answer is refused entirely with a message directing clinicians to primary sources.

Layer 4: Direct Advice Detection uses regex patterns to catch prescriptive language: "you should administer," "give the patient X mg," "the recommended dose is." These trigger a safety flag regardless of confidence score, because a retrieval system should surface evidence, not prescribe treatment.

Production-Grade Evaluation

Standard RAG metrics (answer relevance, context precision) are necessary but insufficient for clinical use. I added clinical-specific evaluation:

Metric Target Achieved
Hallucination rate < 2% 0.0%
Faithfulness (n-gram overlap) > 0.85 0.85+
Answer relevance > 0.80 0.80+
Context precision > 0.85 0.85+
Safe refusal rate Acceptable ~15%

The 15% safe refusal rate is intentional. Rather than answering with low confidence, the system refuses queries it can't ground in sources. For clinical settings, this is the correct tradeoff: it's far better to say "I don't have enough evidence" than to fabricate.

Technical Decisions

Why Local LLM Inference (Ollama)

All LLM calls are processed locally via Ollama running on the same machine. Patient data and clinical queries never leave the network. This is non-negotiable for healthcare deployments, and it eliminates vendor lock-in and API cost concerns for at-scale deployments.

Why Hybrid Search

Pure vector search fails on clinical abbreviations and exact-match queries. Reciprocal Rank Fusion combines pgvector cosine similarity with BM25 keyword matching. This matters because "factor V Leiden" and "factor VIII" are clinically distinct but semantically similar. The hybrid approach catches both semantic and lexical matches.

Why PubMedBERT

PubMedBERT (microsoft/BiomedNLP-BiomedBERT-base-uncased-abstract) is trained on biomedical literature and dramatically outperforms general-purpose embeddings on clinical text. The performance gap is largest on abbreviation-heavy queries and domain-specific terminology.

Why Low Temperature (0.1)

Temperature 0.1 produces deterministic, reproducible generation. It sacrifices prose fluency for accuracy -- clinical correctness beats readability. Creative hallucination is minimized at the cost of more formulaic answers.

Why Conservative Chunking (200 tokens, 50 overlap)

Smaller chunks mean more precise retrieval at the cost of less context per chunk. The 50-token overlap prevents information loss at chunk boundaries. Larger chunks reduce retrieval precision and increase hallucination risk.

What I Learned

This project revealed the gap between "RAG demo" and "clinically deployable RAG." Healthcare organizations typically need help in four areas:

Evaluation Infrastructure: Most teams can stand up a RAG pipeline in a week. Few have the infrastructure to measure whether it's safe. Building domain-specific evaluation sets, implementing sentence-level grounding checks, and establishing meaningful safety thresholds requires both clinical domain expertise and ML engineering.

Safety Engineering: Off-the-shelf guardrails (content filters, toxicity detectors) are designed for consumer AI. Clinical safety requires domain-specific mechanisms: grounding verification against medical literature, confidence calibration for clinical questions, and detection of prescriptive language that crosses the line from information to advice.

Regulatory Navigation: The FDA's guidance on Clinical Decision Support Software defines four criteria determining whether a system is regulated as a medical device. Designing a system that stays within non-device classification while remaining clinically useful requires understanding both technical and regulatory constraints.

Deployment Complexity: Running a clinical RAG system in production means HIPAA-compliant infrastructure, audit logging, access controls, and monitoring. The ML pipeline is often the easy part. The compliance and operations layer is where most projects stall.
