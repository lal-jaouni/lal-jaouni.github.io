PlateHandler Motion Control

A 5-axis embedded motion control system for laboratory plate handling automation. Custom firmware on an ESP32 with H-Bot kinematics, real-time collision detection, and a desktop operator interface — from stepper drivers to GUI.

Role Sole Developer & Architect
Organization Meso Scale Diagnostics
Scale ~4,500 lines of code
Platform ESP32 / C++ & Python
Axes 5 (X, Y, Z, Theta, Gripper)

The Problem

Laboratory plate handling sounds simple until you actually try to automate it. You need precise XY positioning (plates have to land exactly in instrument nests), Z-axis control for lifting and lowering, a gripper that can rotate plates to different orientations, and all of this has to happen without crashing into anything. Off-the-shelf solutions either cost too much or don't fit the specific geometry of the instrument layout.

What I Built

5-Layer Architecture

Designed the firmware as a clean five-layer stack: low-level stepper drivers at the bottom, then axis abstraction, kinematics engine, motion planning, and high-level command interface at the top. Each layer only talks to its neighbors, making it possible to swap out hardware (different motors, different drivers) without touching the motion logic.

H-Bot Kinematics

The XY stage uses an H-Bot belt configuration where two motors work together to produce planar motion. I implemented the forward and inverse kinematics transformations with configurable motor inversion and per-axis calibration (steps/mm), so the system can be tuned to any mechanical setup.

Theta-Gripper Coupling

The gripper and rotation axes are mechanically coupled — rotating the plate also changes the jaw opening. I modeled this relationship (jaw = Gmm - C × Θdeg) and built a coupling engine that compensates automatically, so operators can rotate and grip independently without worrying about the mechanical linkage.

Safety & Collision Detection

Built a multi-level safety system: hardware limit switches at the physical boundary, software limits for the configured workspace, polygon-based boundary validation using a ray-casting algorithm, and trajectory pre-validation that checks the entire planned path before any motor moves. If anything fails at any level, the system stops.

Motion Planning

Implemented trapezoidal velocity profiling with configurable acceleration and deceleration ramps. A producer-consumer command queue ensures deterministic execution order, and per-axis speed controls allow global and individual overrides for different operation types.

Operator Interface

Built a PyQt5 desktop GUI with real-time position feedback, jog controls for each axis, serial communication management, and a command interface for scripted operations. The async I/O architecture keeps the UI responsive during long motion sequences.

What This Taught Me

Embedded work forces a different kind of discipline. There's no debugger stepping through a motor in motion, no rollback on a collision. Every motion command has to be validated before it executes, and the safety system has to work at every level independently. This project gave me a deep appreciation for layered abstraction — the same firmware runs unchanged whether you're jogging an axis by hand or executing a scripted pick-and-place sequence, because the command interface doesn't know or care what's above it.
